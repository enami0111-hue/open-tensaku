# Motivation Article Writer (v2)
 
「大学名+志望理由」「大学名+学部学科+志望理由」のSEO記事を執筆するためのスキル。
 
## 発動条件
 
以下のいずれかに該当する依頼で発動する：
- 「○○大学の志望理由記事を書いて」
- 「○○大学○○学部の志望理由を作成」
- 「志望理由記事の量産」
- 「大学受験のSEO記事」
---
 
## 絶対原則（NEVER VIOLATE）
 
### 原則1: 一次情報なき執筆は禁止
 
`university-research`スキルで収集した以下の一次情報がなければ、執筆を開始しない。
 
- アドミッションポリシー（求める学生像）
- カリキュラム（必修科目・選択科目・特色科目）
- 著名な教授・研究室・研究テーマ（最低3つ）
- 学部独自のプログラム（留学・インターン・PBL等）
- 卒業後の進路実績（業界別・職種別）
- **学部の最新改革情報**（過去5年以内のカリキュラム改訂・必修化・新プログラム）
これらが揃っていない場合、ユーザーに `university-research` を先に実行するよう促す。
 
### 原則2: テンプレ感の徹底排除
 
以下の表現を**絶対に使わない**：
 
- 「○○学部では、政治と経済を幅広く学べます」（抽象論）
- 「貴学のアドミッションポリシーに共感し」（コピペ感）
- 「将来は社会に貢献したい」（誰でも書ける）
- 「グローバルな視点を身につけたい」（具体性ゼロ）
代わりに、**固有名詞**を必ず含める：
- 教授名、研究室名、科目名、書籍名、論文名、プログラム名、提携大学名
### 原則3: 親の進路相談室「3原則」を適用
 
執筆する記事内の例文・モデル志望理由は、すべて以下の3原則を満たす：
 
1. **結論ファースト**: 冒頭1文で「何をしたいか」が明確
2. **経験→だから学びたい**: 抽象的な憧れではなく、具体的経験から学習動機を導出
3. **抽象を具体に**: 「グローバル人材」ではなく「東南アジアのマイクロファイナンス現場で働く実務家」
### 原則4: 断定の根拠化（v2新設・E-E-A-T担保）
 
「〜が評価されます」「〜が求められます」と**断定する場合は、必ず根拠を併記**する。
 
NGパターン：
- 「実質的に評価されるのは○○です」（根拠なし）
- 「合格者は皆○○を書いています」（データなし）
OKパターン：
- 「アドミッションポリシーから読み取れる**仮説**として、○○が重視されていると考えられます」
- 「○○大学公式の合格者インタビュー（出典URL）では、○○への言及が頻出しています」
- 「これは合格者の傾向から見えてくる**暗黙の評価軸**です」
根拠なき断定は**Googleの品質評価ガイドラインで低品質判定**されるリスクがある。
 
### 原則5: 受験生タイプ別の論点提示（v2新設）
 
「○○の固有論点を3つ全部書け」ではなく、「**3つの典型的な志望者タイプ別に、刺さる論点を1つ選ばせる**」構造で書く。
 
理由：
- 異なるタイプの論点を1人の受験生が全部詰め込むと、**人物像が分裂**する
- 受験生が「自分はどのタイプか」を考えながら読むことで、**滞在時間・回遊率が向上**
- 添削サービスへの誘導も「自分のタイプに合った深掘り」として自然に訴求できる

## 記事構造（v2：必ずこの順序で執筆）
 
### 1. タイトル（H1）
 
形式：`【○○大学○○学部】志望理由の書き方｜合格者が押さえた3つの固有論点`
 
ポイント：
- 「○○大学 ○○学部 志望理由」のキーワード完全一致
- 「3つの固有論点」で他記事との差別化を訴求
- 文字数32文字以内（SERP切れ防止）
### 2. 導入（300字以内に圧縮 ※v1の半分）
 
構成：
1. 検索者の悩みへの共感（「他大学の志望理由テンプレを流用していませんか？」）
2. この記事独自の価値提示（学部固有の論点に踏み込む）
3. 柱記事Aへの誘導（「基本構造は柱Aで、本記事は○○大学への適用に絞る」）
### 3. ★冒頭サマリーボックス（v2新設・離脱率対策）
 
導入直後に、**この記事の結論を箇条書きで先出し**する。
 
形式：
```markdown
> **この記事の結論**
> - ○○学部の志望理由は「3つの固有論点」のうち**自分に合う1つ**を深掘りすべき
> - 論点A（学際派向け）: [固有論点1]
> - 論点B（探究派向け）: [固有論点2]
> - 論点C（スキル派向け）: [固有論点3]
> - 失敗パターン3つ＋自己チェック10項目も後半で紹介
```
 
この構造により、スマホで読む受験生がスクロール1〜2回で全体像を把握できる。
 
### 4. H2-1: ○○学部のアドミッションポリシーと「本当に求められる学生像」
 
- アドミッションポリシーの引用（必ず一次情報URLを併記）
- 表面的な文言の裏にある「実質的に評価される資質」の解説
- **必ず原則4を適用**：断定ではなく「仮説」「傾向」として提示
### 5. H2-2: ○○学部の志望理由で押さえるべき3つの固有論点【例文あり】
 
★v2の最重要セクション。**タイプ別構造**で書く。
 
3つの論点は、必ず以下の構造で書く：
 
```markdown
### 論点A（[タイプ名]向け）: [固有のプログラム/研究/カリキュラム名]
 
**こんな受験生に刺さる**: [どんな関心・志向を持つ受験生か]
 
**なぜこれが固有論点か**: 他大学にはない/極めて少ない理由を100字程度で説明
 
**Before例（よくある凡庸な書き方）**: 100字程度
**After例（固有名詞を加えた改善版）**: 200字程度
**Before→Afterの差分解説**: 何を変えたかを30字程度で明示
```
 
**3タイプの分類例（学部によって調整）**：
 
| タイプ | 志向 | 適する論点の方向性 |
|---|---|---|
| 学際派 | 専攻に縛られず横断的に学びたい | カリキュラム・コース選択の自由度 |
| 探究派 | 特定テーマを深く研究したい | 教員・ゼミ・留学プログラム |
| スキル派 | 具体的な能力・資格を獲得したい | 必修科目・新設プログラム・産学連携 |
 
学部の特性によっては「実務派」「研究者派」「グローバル派」など別の3タイプ分類も可。**重要なのは「分裂しない3つのタイプ」を作ること**。
 
論点の選び方：
- 論点A: カリキュラム・科目レベルの固有性
- 論点B: 研究室・教授・研究プログラムレベルの固有性
- 論点C: 制度・新設プログラム・最新改革レベルの固有性（**過去5年以内の新要素を必ず含める**）
### 6. H2-3: ○○学部の志望理由でよくある失敗例3パターンと改善方法
 
「悪い例→良い例」を3パターン紹介。各パターンは以下の構造：
 
```markdown
### 失敗パターン1: [パターン名]
 
**悪い例**: 100字程度の典型的な悪い例
**なぜダメか**: 30字程度の指摘
**改善例**: 100字程度の改善版
```
 
失敗パターンの定番3つ：
1. 「貴学の理念に共感」型（コピペ感）
2. 「将来○○になりたい」型（学部接続なし）
3. 「○○を学びたい」型（具体性なし）
### 7. H2-4: 入試方式別の志望理由書き分け（総合型選抜・指定校推薦・一般選抜）
 
入試方式ごとに重視されるポイントが異なる：
 
- **総合型選抜**: 主体性・経験・探究テーマとの接続
- **指定校推薦**: 高校での活動実績との一貫性
- **一般選抜**: 出願書類は形式的なため、面接・小論文での口頭説明準備
各入試の**具体的な書類項目**（字数制限・設問内容）も一次情報から記載する。
 
### 8. H2-5: ○○学部志望理由の自己チェックリスト10項目
 
10項目程度のチェックリスト形式：
- [ ] 学部固有の科目名・教授名・プログラム名が3つ以上含まれているか
- [ ] 「グローバル」「社会貢献」などの抽象語を5回以上使っていないか
- [ ] 高校時代の具体的経験から学習動機を導出しているか
- [ ] 競合学部（同分野の他大学）と比較した上で、当該学部を選ぶ理由が示されているか
- ...
### 9. まとめ（150〜200字）
 
- 学部固有性の重要性を再強調
- 「3つの論点のうち、自分のタイプに合う1つを深掘り」という方針を再確認
- 同分野の他大学記事への内部リンク

## 文字数・密度のルール
 
- 全体: 5,000〜8,000字（網羅性重視）
- 各H2セクション: 600〜1,500字
- 例文は必ず**Before/Afterセット×3つ**含める
- 固有名詞（教授名・科目名・プログラム名）は最低15個含める
- **過去5年以内の最新改革情報**を最低1つ含める

## H2見出しのSEOルール（v2強化）
 
H2見出しには**ロングテールキーワードを必ず埋め込む**。「3つの固有論点」だけではなく、検索キーワードを含む形で書く。
 
| NG（キーワード性なし） | OK（ロングテール含む） |
|---|---|
| 「3つの固有論点」 | 「○○学部の志望理由で押さえるべき3つの固有論点【例文あり】」 |
| 「失敗パターン」 | 「○○学部志望理由でよくある失敗例3パターンと改善方法」 |
| 「自己チェック」 | 「○○学部志望理由の自己チェックリスト10項目」 |
 
これにより以下のロングテールキーワードを同時に取りに行く：
- 「○○学部 志望理由 例文」
- 「○○学部 志望理由 失敗」
- 「○○学部 志望理由 添削」
- 「○○学部 志望理由 書き方」

## 内部リンク設計（v2強化）
 
### 必須内部リンク（最低5つ）
 
各記事から以下にリンクする：
 
1. **柱記事A「3原則完全ガイド」**（権威性担保）
2. **同大学の他学部記事**（横展開）
3. **同分野の他大学記事**（比較検討） ※必ず3校以上にリンク
4. **入試方式別の柱記事C「総合型選抜ガイド」**
5. 添削サービスLP（CV導線）※後で追加
### v2新設：内部リンク運用ルール
 
PoC段階では同分野の他大学記事がまだ存在しないため、**公開順序**を戦略的に設計する：
 
```
公開順序の原則：
1. 柱記事A（3原則完全ガイド）を最優先公開
2. 同分野の量産記事を「同時または近接公開」
3. 後続記事公開時に、先行記事の内部リンクを必ず追記更新
```
 
例：国際系学部4記事を量産する場合
```
Day 1: 柱記事A公開
Day 7: 早稲田SILS、上智FLA、立教GLAP、ICU を同日公開
       → 4記事すべてに相互リンクを最初から実装
```
 
これにより、リンク切れ状態を回避し、リンクジュースの流れを最初から確保できる。
 
### アンカーテキストのルール
 
- 完全一致を避ける（過剰最適化判定リスク）
- 1記事あたりの内部リンクは10〜15が上限
- 自然な文脈での誘導（「○○の比較検討に：他の主要大学の志望理由はこちら」等）

## 一次情報の引用位置（v2新設）
 
E-E-A-T担保のため、参考URLを**記事末尾にまとめる**のではなく、**本文中で文脈に沿って都度引用**する。
 
NG（v1の悪い例）：
```markdown
[本文すべて]
---
## 参考一次情報
- 早稲田大学公式: https://...
- カリキュラム: https://...
```
 
OK（v2推奨）：
```markdown
本文中で「（早稲田大学公式サイト『学部長挨拶』より引用）」と都度リンク
末尾には「主要参考情報」として最重要URL3〜5つに絞ってまとめる
```

## 専門用語の表記ルール（v2新設）
 
英語の正式名称が長い場合（10字以上）、初出時に**意訳での簡潔説明**を優先する。
 
NG例：
> APMプログラム（Area Studies and Plurilingual/Multicultural Education Program）
 
OK例：
> APMプログラム（4言語＋地域研究の統合プログラム、2017年スタート）
 
正式英語名は記事末尾の「用語集」セクションにまとめる。本文では受験生が**読み流せる簡潔な説明**を優先する。

## 品質チェック（v2強化・出力前に必ず実行）
 
執筆完了後、以下を自己点検し、不合格項目があれば修正：
 
### 内容品質
1. ✅ 固有名詞（教授・科目・プログラム）が15個以上含まれているか
2. ✅ 過去5年以内の最新改革情報が含まれているか
3. ✅ 3つの固有論点それぞれに、競合記事にない独自の切り口があるか
4. ✅ 例文が**Before/After形式**（v2新設）で書かれているか
5. ✅ 例文が3原則（結論ファースト・経験→学び・抽象を具体に）を満たすか
6. ✅ 抽象語（グローバル・社会貢献・幅広く）を5回以上使っていないか
7. ✅ 同分野の競合大学（最低2校）と**比較**して論じているか
### 構造品質
8. ✅ 冒頭サマリーボックス（v2新設）が導入直後に配置されているか
9. ✅ 3つの論点が**タイプ別**（v2新設）に分類されているか
10. ✅ H2見出しに**ロングテールキーワード**（v2強化）が含まれているか
### E-E-A-T担保
11. ✅ 断定表現に根拠が併記されているか（v2新設）
12. ✅ 一次情報のURLが**本文中で都度引用**（v2新設）されているか
13. ✅ 一次情報のURL引用が3つ以上あるか
### SEO
14. ✅ タイトル32字以内、ディスクリプション120字以内か
15. ✅ メインキーワード密度が0.5〜1.5%か

## アンチパターン（やってはいけない執筆）
 
- ❌ 大学公式サイトの文言をそのまま貼り付ける（コピペ判定リスク）
- ❌ 他大学の記事から固有名詞だけ差し替える（重複コンテンツ判定リスク）
- ❌ 「志望理由の書き方」一般論を冗長に書く（検索意図不一致）
- ❌ 「3原則」を冒頭で長々と説明する（柱記事に内部リンクで処理）
- ❌ ChatGPT臭のする文体（「〜していきましょう」「いかがでしょうか」連発）
- ❌ 3つの論点を「全部書け」と読める書き方（v2新設・タイプ別に分類すべき）
- ❌ 例文が完成形のみで、書き方プロセスが見えない（v2新設・Before/After必須）
- ❌ 根拠なき断定（v2新設・「〜と考えられる」「〜の傾向がある」を使う）
- ❌ 英語正式名称をそのまま列挙（v2新設・意訳での簡潔説明を優先）

## v1からv2への主な変更点
 
| 項目 | v1 | v2 |
|---|---|---|
| 3つの論点の構造 | 並列で全部書く | タイプ別に1つ選ばせる |
| 例文 | 完成形のみ | Before/After形式 |
| 冒頭 | 導入のみ | サマリーボックス追加 |
| 断定表現 | 制約なし | 根拠併記必須 |
| H2見出し | キーワード性弱い | ロングテール埋込必須 |
| 一次情報URL | 末尾まとめ | 本文中で都度引用 |
| 内部リンク | 想定先記載のみ | 公開順序の運用ルール明記 |
| 最新情報 | 任意 | 過去5年以内の改革情報必須 |
| 英語専門用語 | 正式名称併記 | 意訳での簡潔説明優先 |
